## 2. 项目背景与产品目标

### 2.1 项目背景

中小企业在资产管理领域普遍面临以下痛点：

- **台账混乱**：资产信息散落在Excel、纸质单据中，缺乏统一电子台账，查询追溯困难；
- **流程缺失**：资产领用、归还、盘点依赖人工，执行效率低、易遗漏；
- **权责不清**：资产归属不明确，部门间流转无记录，离职交接时资产流失频发；
- **盘点困难**：定期盘点依赖人工核对，盘盈盘亏差异难以跟踪处理；
- **数据孤岛**：资产数据与财务、采购系统割裂，管理层无法实时掌握资产全貌；
- **操作门槛高**：非专业人员难以使用复杂筛选，资产管理员成为信息瓶颈。

### 2.2 产品目标

EAMS一期为中小企业构建**轻量化、可落地**的资产管理数字化底座：

1. **统一资产台账**：资产入库到领用/归还全流程数字化；
2. **规范领用归还**：申请→审批→登记→追溯闭环；
3. **支撑盘点作业**：任务创建→执行→差异记录→处理；
4. **AI智能查询**：集成DeepSeek，自然语言检索资产；
5. **RBAC权限体系**：超级管理员、资产管理员、部门管理员、普通员工四级隔离。

### 2.3 产品定位

- **一期定位**：MVP，覆盖资产"管、借、盘、查"核心场景
- **目标上线周期**：8-10周

## 3. 术语定义

| 术语 | 英文 | 定义 |
|------|------|------|
| 资产 | Asset | 单位价值≥2000元、使用年限≥1年的有形固定资产（IT设备、办公家具、生产设备等） |
| 资产台账 | Asset Ledger | 记录每项资产全生命周期信息的电子档案 |
| 资产编码 | Asset Code | 系统自动生成，格式：`AS-{类别码}-{年月}-{4位流水}`，如 `AS-IT-2606-0001` |
| 盘盈 | Inventory Surplus | 实际数量/价值大于账面记录 |
| 盘亏 | Inventory Shortage | 实际数量/价值小于账面记录 |
| 折旧 | Depreciation | 平均年限法按月计提 |
| 净残值率 | Residual Rate | 默认 **5%** |
| AI智能查询 | AI Smart Query | 基于DeepSeek的自然语言资产检索 |
| 软删除 | Soft Delete | `is_deleted=1`，记录不物理删除 |
| 领用 | Requisition | 员工申请使用闲置资产 |
| 归还 | Return | 将已领用资产交回，恢复闲置状态 |

## 5. 全局通用规则

### 5.1 分页规范

| 参数 | 类型 | 必填 | 默认值 | 说明 |
|------|------|:----:|--------|------|
| `pageNum` | Integer | 否 | 1 | 最小值1 |
| `pageSize` | Integer | 否 | 10 | 可选 10/20/50/100，最大100 |

后端统一返回：

```json
{
  "code": 200,
  "message": "操作成功",
  "data": {
    "total": 100,
    "pageNum": 1,
    "pageSize": 10,
    "pages": 10,
    "list": []
  }
}
```

### 5.2 统一返回体

```json
{ "code": 200, "message": "操作成功", "data": {} }
```

| code | HTTP状态码 | 说明 |
|------|:----------:|------|
| 200 | 200 | 操作成功 |
| 400 | 400 | 参数校验失败，`message` 返回具体原因 |
| 401 | 401 | 未登录或Token过期，前端跳转登录页 |
| 403 | 403 | 无权限，提示"您没有权限执行此操作" |
| 404 | 404 | 资源不存在 |
| 429 | 429 | 请求过于频繁 |
| 500 | 500 | 服务器内部错误，提示"系统繁忙，请稍后重试" |

**后端注意**：`message` 字段直接返回面向用户的友好提示文本，前端用 `ElMessage` 展示，不要在 message 中返回技术异常堆栈。

### 5.3 文件上传限制

| 限制项 | 限制值 | 超出提示 |
|--------|--------|----------|
| 单文件最大 | 10MB | "文件大小不能超过10MB" |
| 单次数量 | 5个 | "单次最多上传5个文件，请分批上传" |
| 允许类型 | jpg/jpeg/png/gif/bmp/pdf/doc/docx/xls/xlsx/ppt/pptx | "不支持的文件类型：{文件名}" |
| 资产图片 | 仅 jpg/jpeg/png，≤2MB | "仅支持 JPG/PNG 格式图片，大小不超过2MB" |

### 5.4 软删除规则

- 所有删除操作均为软删除（`is_deleted=1`）；
- 查询接口默认过滤已删除记录；
- 关联约束：**已被领用（在用/借用）的资产不可删除**，需先归还。

#### 5.7.2 过期时长规范

| 数据类别 | 过期时间 | 说明 |
|----------|----------|------|
| 系统参数 | **持久**（无过期） | 参数编辑时主动更新对应 field，应用重启从 MySQL 重新加载 |
| 字典缓存 | **24 小时** | 字典变更时主动清除，24 小时兜底 |
| 资产详情 | **30 分钟** | 资产编辑后主动清除 |
| 资产列表 | **5 分钟** | 分页查询结果，数据变化频繁 |
| Token 黑名单 | **与 Token 剩余有效期一致** | 超过 Token 自然过期则自动清理 |
| 登录失败计数 | **30 分钟** | 与锁定时长一致 |
| 登录锁定标记 | **30 分钟** | 到期自动解锁 |
| 分布式锁 | **最大 30 秒** | 防止死锁，业务执行完主动释放 |
| AI 分钟限流 | **60 秒** | 滑动窗口，每分钟重置 |
| AI 小时限流 | **3600 秒** | 滑动窗口，每小时重置 |
| AI 日调用计数 | **当日 23:59:59 剩余秒数** | 每日零点重置 |
| AI 降级标记 | **5 分钟** | 到期后自动探测恢复 |

#### 5.7.3 缓存更新策略

| 策略 | 适用场景 | 说明 |
|------|----------|------|
| **主动清除** | 字典、资产详情 | 数据变更时删除对应 Key，下次查询重新加载（Cache-Aside） |
| **自然过期** | 资产列表、限流计数 | 到期自动清除，容忍短期数据不一致 |
| **启动预加载** | 全量字典 | 应用启动时从 MySQL 加载写入 Redis，设置 24h 过期 |
| **写入穿透保护** | 所有查询缓存 | 查询不存在的数据时缓存空值（Null Object），过期时间 **1 分钟**，防缓存穿透 |

#### 5.7.4 分布式锁标准

基于 **Redisson** 或 **Lettuce `SET NX EX`** 实现，统一规范：

| 配置项 | 值 | 说明 |
|--------|-----|------|
| 锁 Key 前缀 | `eams:lock:{module}:{bizKey}` | 见 5.7.1 Key 命名 |
| 最大持有时间 | **30 秒** | 超时自动释放，防死锁 |
| 等待获取时间 | **3 秒** | 获取不到锁时最多等待 3 秒 |
| 获取失败策略 | 返回错误提示 | 不无限重试，提示用户"操作繁忙，请稍后重试" |
| 释放原则 | **谁加锁谁释放** | 通过 UUID 校验锁归属，防误删他人锁 |

**分布式锁 Java 调用规范**：

```java
// 推荐方式：使用 RedisUtil 封装方法
String lockKey = "eams:lock:requisition:" + assetId + ":" + userId;
String lockValue = RedisUtil.tryLock(lockKey, 3, 30); // 等待3s，持锁30s
if (lockValue == null) {
    throw new BusinessException("操作繁忙，请稍后重试");
}
try {
    // 业务逻辑
} finally {
    RedisUtil.unlock(lockKey, lockValue);
}
```

### 5.8 数据库通用审计字段规范
系统**全部业务数据表**（用户、资产、领用、盘点、字典、AI日志等）强制增加5个通用字段，无例外：

| 字段名 | 类型 | 含义 | 填充规则 | 业务说明 |
|--------|------|------|----------|----------|
| create_time | datetime | 创建时间 | 新增记录时自动填充当前时间，不可修改 | 记录数据首次入库时间 |
| update_time | datetime | 更新时间 | 新增/编辑时自动刷新当前时间 | 每次修改自动更新，不可手动传参 |
| create_by | bigint | 创建人ID | 新增时自动填入当前登录用户ID | 关联sys_user.id，追溯数据创建操作者 |
| update_by | bigint | 修改人ID | 新增/编辑时自动填入当前登录用户ID | 追溯最后一次修改操作者 |
| is_deleted | tinyint(1) | 有效标志（软删除） | 默认值 0 | 0=正常有效数据；1=已逻辑删除，所有查询接口默认过滤 `is_deleted=1` 数据 |

#### 配套开发约束
1. 前端提交新增/编辑表单时，**禁止传递以上5个字段**，统一由后端拦截器/MyBatis-Plus自动填充；
2. 逻辑删除规则：业务删除操作仅更新 `is_deleted=1`，不执行物理DELETE；
3. 关联查询约束：资产、领用、盘点等关联业务查询，必须携带 `is_deleted=0` 过滤条件；
4. 历史归档场景：如需查询已删除数据，后台管理接口单独增加「包含已删除」筛选开关，默认关闭。

| 场景 | code | message |
|------|------|---------|
| 用户名或密码错误 | 400 | "用户名或密码错误"（不区分用户名/密码错误） |
| 账号已禁用 | 403 | "账号已被禁用，请联系管理员" |
| 账号已锁定 | 403 | "账号已被锁定，请{m}分钟后重试"（m=剩余分钟数） |
| 参数为空 | 400 | "请输入用户名和密码" |
| 登录成功 | 200 | "登录成功" |

**Redis 实现**：

| 功能 | Redis Key | 实现逻辑 |
|------|-----------|----------|
| 登录失败计数 | `eams:login:fail:{username}` | 每次密码错误 `INCR`，首次设置 `EXPIRE 1800`（30分钟）；达到5次触发锁定 |
| 账号锁定标记 | `eams:login:lock:{username}` | 失败达5次后 `SET lock:username EX 1800`，登录时先检查此 Key 是否存在 |
| Token 黑名单 | `eams:token:blacklist:{tokenId}` | 主动登出或强制下线时，将 Token 的 JTI 写入黑名单，过期时间 = Token 剩余有效期 |
| Token 黑名单校验 | — | 每次请求在 JWT 过滤器中先查黑名单，命中则返回 401 `"登录已过期，请重新登录"` |

```java
// 登录失败计数伪代码
String failKey = "eams:login:fail:" + username;
int attempts = RedisUtil.incr(failKey, 1800); // INCR + 首次设过期
if (attempts >= 5) {
    RedisUtil.set("eams:login:lock:" + username, "1", 1800);
    RedisUtil.del(failKey); // 锁定后清除计数
    throw new BusinessException(403, "账号已被锁定，请" + remainingMin + "分钟后重试");
}
// 登录成功后清除失败计数
RedisUtil.del(failKey);
```

| 全量字典缓存 | `eams:dict:all` | 应用启动时（`@PostConstruct`）从 MySQL 加载全量字典，写入 Redis Hash，过期 24h |
| 单字典缓存 | `eams:dict:{dictCode}` | 按字典编码分别缓存（Hash 结构：field=dictValue, value=dictLabel），用于下拉框查询 |
| 缓存更新 | — | 字典类型或字典项增删改后，调用 `RedisUtil.del("eams:dict:all")` + `RedisUtil.del("eams:dict:" + dictCode)` 双清 |
| 查询流程 | — | 先查 Redis → 命中直接返回 → 未命中查 MySQL → 写入 Redis 并设过期 → 返回 |
| 空值防护 | — | MySQL 查不到时缓存空值 `{}`，过期 1 分钟，防缓存穿透 |

```java
// 字典查询伪代码
public List<DictItem> getDictItems(String dictCode) {
    String cacheKey = "eams:dict:" + dictCode;
    Map<String, String> cached = RedisUtil.hgetAll(cacheKey);
    if (cached != null) {
        return cached.isEmpty() ? Collections.emptyList() : toDictItemList(cached);
    }
    List<DictItem> items = dictMapper.selectByCode(dictCode);
    if (items.isEmpty()) {
        RedisUtil.hset(cacheKey, "__empty__", "1", 60); // 空值缓存1分钟
    } else {
        Map<String, String> map = items.stream().collect(toMap(DictItem::getValue, DictItem::getLabel));
        RedisUtil.hmset(cacheKey, map, 86400); // 24小时
    }
    return items;
}
```

**业务规则**：
1. 系统预置参数**不可删除**（paramKey 固定），但可编辑参数值；
2. 参数修改后**立即生效**——后端清除对应 Redis 缓存，下次读取加载新值；
3. 禁用状态的参数不生效，系统自动回退到默认值；
4. 参数值修改均记录操作日志 + 变更历史；
5. 数字类型参数后端需校验范围（如锁定次数 1-20 次，锁定时长 5-1440 分钟）。

**后端注意**：
- 应用启动时（`@PostConstruct`）从 MySQL 加载全部启用参数写入 Redis
- 每个业务模块通过 `ConfigService.getString(key)` / `ConfigService.getInt(key)` 等统一方法读取，不直接查 MySQL

**Redis 实现**：

| 功能 | Redis Key | 实现逻辑 |
|------|-----------|----------|
| 全量参数缓存 | `eams:config:all` | Hash 结构，field=paramKey, value=paramValue；应用启动时加载，参数编辑后更新对应 field |
| 参数编辑生效 | — | 编辑保存后，调用 `RedisUtil.hset("eams:config:all", paramKey, newValue)` 实时更新缓存 |
| 参数禁用回退 | — | 禁用某个参数后，`hset` 将其值设为默认值；前端查询时若 Key 不存在（禁用/新参数未加载），读取 default_value |
| 过期策略 | — | 无过期时间（持久缓存），依赖编辑操作主动更新；应用重启时从 MySQL 重新加载 |

```java
// 配置读取伪代码
public class ConfigService {
    public String getString(String key) {
        String value = RedisUtil.hget("eams:config:all", key);
        if (value != null) return value;
        // 缓存未命中，查 MySQL
        SysConfig config = configMapper.selectByKey(key);
        if (config != null && config.getStatus() == 1) {
            RedisUtil.hset("eams:config:all", key, config.getParamValue());
            return config.getParamValue();
        }
        return config != null ? config.getDefaultValue() : null;
    }

    public int getInt(String key) {
        return Integer.parseInt(getString(key));
    }

    public void refresh(String key, String newValue) {
        configMapper.updateValue(key, newValue);
        RedisUtil.hset("eams:config:all", key, newValue);
        // 记录变更历史 + 操作日志
    }
}
```

```java
// 资产列表查询伪代码
public PageResult<AssetVO> listAssets(AssetQueryDTO query) {
    String cacheKey = "eams:asset:list:" + DigestUtils.md5Hex(JSON.toJSONString(query));
    String cached = RedisUtil.get(cacheKey);
    if (cached != null) {
        return JSON.parseObject(cached, PageResult.class);
    }
    PageResult<AssetVO> result = assetMapper.selectPage(query);
    RedisUtil.set(cacheKey, JSON.toJSONString(result), 300); // 5分钟
    return result;
}
```

#### 6.2.4 折旧计算

**公式（平均年限法）**：

```
月折旧额 = 原值 × (1 - 残值率) ÷ (年限 × 12)
累计折旧 = 月折旧额 × 已提月数
净值 = 原值 - 累计折旧
```

**折旧明细弹窗**：

| 列 | 字段 |
|----|------|
| 计提月份 | `depreciation_month` |
| 月折旧额 | `monthly_amount` |
| 累计折旧 | `accumulated` |
| 净值 | `net_value` |
| 状态 | 已计提/待计提 |

**规则**：
- 从采购日期**次月**开始计提
- 净值 ≤ 原值 × 残值率 时停止计提
- 已报废资产停止计提
- 系统每月1日定时任务执行（需后端实现 `@Scheduled`）

```java
try {
    // 1. 检查资产是否闲置
    // 2. 检查是否已有待审批申请
    // 3. 插入领用申请记录
} finally {

}
```

```java
// 盘点任务创建防重伪代码
String lockKey = "eams:lock:inventory:create:" + scopeType + ":" + scopeValue;
String lockValue = RedisUtil.tryLock(lockKey, 3, 30);
if (lockValue == null) {
    throw new BusinessException(400, "操作繁忙，请稍后重试");
}
try {
    // 1. 检查同范围是否有进行中的任务
    // 2. 创建盘点任务
    // 3. 批量更新资产状态为"盘点中"
    // 4. 清除资产列表缓存
} finally {
    RedisUtil.unlock(lockKey, lockValue);
}
```

#### 6.5.3 限流降级策略

**限流维度**（基于 Redis 计数器实现）：

| 维度 | Redis Key | 计数器 | 过期时间 | 限制值 | 超出响应 |
|------|-----------|--------|----------|--------|----------|
| 单用户/分钟 | `eams:ai:rate:min:{userId}` | `INCR` + 首次设过期 | 60s | 10次 | HTTP 429, `"查询过于频繁，请稍后再试"` |
| 单用户/小时 | `eams:ai:rate:hour:{userId}` | `INCR` + 首次设过期 | 3600s | 100次 | HTTP 429, `"今日查询次数已达上限，请明日再试"` |
| 全系统/分钟 | `eams:ai:rate:sys:min` | `INCR` + 首次设过期 | 60s | 100次 | HTTP 429, `"系统繁忙，请稍后重试"` |
| DeepSeek 日调用 | `eams:ai:rate:day` | `INCR` + 首次设当日剩余秒数 | 当日 23:59:59 | 5000次 | 触发降级 |

**Redis 限流统一 Key 前缀**：`eams:ai:rate:`，所有 AI 限流相关计数器均使用此前缀。

**降级标记**：

| Key | 值 | 过期 | 说明 |
|-----|-----|------|------|
| `eams:ai:degraded` | `1` | 300s（5分钟） | 存在此 Key 表示 AI 服务已降级 |

**限流拦截伪代码**：

```java
public void checkAiRateLimit(Long userId) {
    // 1. 检查降级标记
    if (RedisUtil.exists("eams:ai:degraded")) {
        throw new BusinessException(503, "AI智能查询服务暂时不可用，请使用传统筛选查询");
    }
    // 2. 分钟限流
    String minKey = "eams:ai:rate:min:" + userId;
    long minCount = RedisUtil.incr(minKey, 60);
    if (minCount > 10) {
        throw new BusinessException(429, "查询过于频繁，请稍后再试");
    }
    // 3. 小时限流
    String hourKey = "eams:ai:rate:hour:" + userId;
    long hourCount = RedisUtil.incr(hourKey, 3600);
    if (hourCount > 100) {
        throw new BusinessException(429, "今日查询次数已达上限，请明日再试");
    }
    // 4. 系统级限流
    long sysCount = RedisUtil.incr("eams:ai:rate:sys:min", 60);
    if (sysCount > 100) {
        throw new BusinessException(429, "系统繁忙，请稍后重试");
    }
    // 5. DeepSeek 日调用限流
    long dayCount = RedisUtil.incr("eams:ai:rate:day", secondsUntilMidnight());
    if (dayCount > 5000) {
        RedisUtil.set("eams:ai:degraded", "1", 300); // 触发降级
        throw new BusinessException(503, "AI智能查询服务暂时不可用，请使用传统筛选查询");
    }
}
```

**降级流程**：
1. DeepSeek 连续3次调用失败 或 日调用超5000次 → 写入 `eams:ai:degraded = 1`，过期 5 分钟
2. 降级后前端 AI 入口展示横幅："AI智能查询服务暂时不可用，请使用传统筛选查询"，输入框禁用
3. `eams:ai:degraded` 过期后自动探测恢复（下次 AI 查询请求时尝试调用 DeepSeek，成功则清除降级标记，失败则重新写入降级标记并重置 5 分钟过期）
4. AI 查询失败不影响其他功能

## 7. 核心业务流程

### 7.1 资产全生命周期总览

```
采购入库 → 领用/借用 → 日常使用 → 归还/调拨
    ↓                      ↓
  维保报修 → 维修处理 → 修复/报废
                ↓
  定期盘点 → 盘盈/盘亏/正常 → 差异处理
                ↓
  折旧计提(每月自动) → 提满/报废停止
                ↓
  报废处置 → 变卖/回收/销毁 → 归档
```

### 7.2 领用流程（固定二级审批）

```
员工提交申请 → 部门管理员审批 → 资产管理员终审 → 资产出库(在用)
     ↓ 驳回          ↓ 驳回            ↓
  通知申请人      通知申请人       通知申请人
                                              ↓
                                    使用完毕 → 归还登记 → 资产恢复(闲置/维修)
```

**一期审批说明**：固定二级审批（部门→资产），无工作流引擎，审批人在"审批管理"页直接通过/驳回。

### 7.3 盘点流程

```
创建任务 → 锁定资产(盘点中) → 逐项核对 → 生成报告
                              ├ 正常 → 确认
                              ├ 盘盈 → 登记入账
                              └ 盘亏 → 登记原因 → 差异待处理
```

### 8.4 Redis 缓存与并发控制

#### 8.4.1 Redis 性能要求

| 指标 | 要求值 | 说明 |
|------|--------|------|
| 缓存命中率 | ≥ 80% | 字典、资产详情的命中率目标 |
| Redis 可用性 | 99.9% | 月度可用性 |
| 单次读写延迟 | ≤ 5ms | Redis 与后端同机房部署 |
| 缓存预热时间 | ≤ 10s | 应用启动时完成字典缓存加载 |
| 最大连接数 | 200 | 连接池 `maxTotal` 配置 |

#### 8.4.2 Redis 容错策略

| 场景 | 策略 | 说明 |
|------|------|------|
| Redis 连接失败 | **自动降级查 MySQL** | Redis 不可用时直接查数据库，不影响业务可用性；记录 ERROR 日志并触发告警 |
| Redis 恢复 | **自动回切** | 每 30 秒探测一次 Redis 连通性，恢复后自动启用缓存 |
| 缓存雪崩 | **过期时间加随机值** | 所有缓存的过期时间在基准值上叠加 ±10% 随机偏移（如 300s → 270~330s），防止集中过期 |
| 缓存击穿 | **分布式锁控制** | 热点 Key 过期时，仅允许一个线程查 MySQL 并回写缓存，其余线程等待锁释放后读缓存 |
| 缓存穿透 | **空值缓存** | 查询不存在的数据时缓存空值（expire=60s），详见 5.7.3 |
| 分布式锁死锁 | **超时自动释放** | 所有锁最大持有 30 秒，通过 `SET NX EX` 原子操作防死锁 |

#### 8.4.3 并发控制

| 场景 | 策略 |
|------|------|
| 领用冲突 | 数据库唯一约束 + Redis 分布式锁双重防护（锁 Key：`eams:lock:requisition:{assetId}:{userId}`） |
| 盘点锁定 | 范围内资产批量更新为"盘点中" + 盘点任务创建分布式锁（锁 Key：`eams:lock:inventory:create:{scopeType}:{scopeValue}`） |
| 资产编码生成 | Redis 分布式锁保护流水号自增（锁 Key：`eams:lock:asset:code:{categoryCode}:{yyMM}`），每日流水独立 |
| 乐观锁 | 资产表 `version` 字段，编辑时校验版本号防并发覆盖（后端返回 `"数据已被他人修改，请刷新后重试"`） |
| AI 限流 | Redis 计数器（INCR + EXPIRE）实现分钟/小时/日三级滑动窗口限流，详见 6.5.3 |

**RedisUtil 工具类说明**：

| 方法 | 参数 | 返回值 | 说明 |
|------|------|--------|------|
| `get(String key)` | key | String / null | 获取字符串值 |
| `set(String key, String value, long expireSec)` | key, value, 过期秒数 | void | 写入字符串并设过期 |
| `del(String key)` | key | void | 删除单个 Key |
| `deleteByPattern(String pattern)` | 正则 pattern（如 `eams:asset:list:*`） | void | 批量模糊删除（使用 SCAN 迭代，不阻塞） |
| `exists(String key)` | key | boolean | 判断 Key 是否存在 |
| `incr(String key, long expireSec)` | key, 过期秒数 | long | 自增 1，首次调用设过期；返回自增后的值 |
| `hgetAll(String key)` | key | Map / null | 获取 Hash 全部字段 |
| `hset(String key, String field, String value, long expireSec)` | key, field, value, 过期秒数 | void | Hash 单字段写入 |
| `hmset(String key, Map<String,String> map, long expireSec)` | key, map, 过期秒数 | void | Hash 批量写入 |
| `tryLock(String key, int waitSec, int holdSec)` | key, 等待秒数, 持有秒数 | String (lockValue) / null | 获取分布式锁，返回 UUID 令牌（用于释放校验）；获取失败返回 null |
| `unlock(String key, String lockValue)` | key, 锁令牌 | void | 释放锁，先校验 lockValue 匹配才删除（Lua 脚本原子操作） |

**使用原则**：
- 所有 Redis 操作通过 `RedisUtil` 统一调用，不直接使用 Jedis/Lettuce 原生 API
- 分布式锁 **必须** 在 `finally` 块中释放，且通过 `lockValue` 校验归属
- `deleteByPattern` 内部使用 `SCAN` 命令分批删除，单次扫描不超过 1000 个 Key
