# 维保报修模块 — 前后端联调核对清单

> 模块版本：V1.0 | 生成日期：2026-06-28

---

## 一、业务流程图

```
员工登录 → [报修登记] → 选择非维修/非报废资产 → 填写故障信息 → 提交
                                                            ↓
                                    repair_order.repair_status = 0（待维修）
                                    asset_info.status → 3（维修）
                                    pre_repair_status = 快照原状态
                                                            ↓
              [维修处理] ← 资产管理员接单 → repair_status = 1（维修中）
                                                            ↓
                         ├── 标记已修复 → repair_status = 2
                         │                  asset_info.status → pre_repair_status（恢复）
                         │
                         └── 标记无法修复 → repair_status = 3
                                            asset_info.status 保持 3（维修）
```

## 二、数据库表结构

### 2.1 repair_order（报修单）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT UNSIGNED | 主键 |
| repair_no | VARCHAR(30) UNIQUE | `WX-YYYYMMDD-XXXX` |
| asset_id | BIGINT UNSIGNED | FK → asset_info.id |
| applicant_id | BIGINT UNSIGNED | 报修人 |
| fault_type | VARCHAR(20) | 硬件故障/软件故障/网络故障/配件更换/其他 |
| urgency | TINYINT | 0-普通, 1-紧急 |
| fault_desc | VARCHAR(500) | 10-500字符 |
| fault_images | VARCHAR(1000) | 图片URL逗号分隔 |
| contact_phone | VARCHAR(20) | 报修人电话 |
| repair_status | TINYINT | **0-待维修, 1-维修中, 2-已修复, 3-无法修复** |
| pre_repair_status | TINYINT | 维修前资产状态快照 |
| remark | VARCHAR(200) | 备注 |

### 2.2 repair_record（维修记录）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT UNSIGNED | 主键 |
| repair_order_id | BIGINT UNSIGNED | FK → repair_order.id |
| repair_method | VARCHAR(20) | 现场维修/送修/上门维修/远程支持 |
| repair_person | VARCHAR(50) | 维修人员 |
| repair_fee | DECIMAL(12,2) | ≥0 |
| start_date | DATE | 开始维修日期 |
| finish_date | DATE | 修复日期 |
| fault_reason | VARCHAR(500) | 故障根因 |
| solution | VARCHAR(500) | 处理措施 |
| repair_files | VARCHAR(1000) | 附件URL逗号分隔 |

### 接口详情

**1. 提交报修**
```

Success:  200 "报修申请已提交，维修人员将尽快处理"

```

          "startDate":"2026-06-28","faultReason":"根因分析10-500字符","solution":"处理措施10-500字符"}
complete需要额外传: "finishDate":"2026-06-29"

## 八、Postman 测试用例

## 九、常见坑点

| # | 问题 | 原因 | 处理 |
|---|------|------|------|
| 1 | `startDate` 传 `"2026-06-28 00:00:00"` 报 `请求参数格式错误` | Jackson 默认只接受 `yyyy-MM-dd` | 已全局配置 `LocalDateDeserializer` 容错截取前10位 |
| 2 | `action` 字段 `@NotBlank` 校验失败 | Controller `setAction` 在 `@Valid` 之后执行 | 已移除 `@NotBlank` |
| 3 | 维修处理按钮可见性不区分角色 | 前端 `v-if` 仅判断 status | DEPT_ADMIN/EMPLOYEE 无菜单权限，侧边栏已过滤 |
| 4 | 紧急程度标红 | `el-table-column` 需用 scoped slot | HandleRepair.vue 已实现 `:class="{'red':row.urgency===1}"` |
