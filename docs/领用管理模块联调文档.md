# 领用管理模块 — 前后端联调核对清单

> 模块版本：V1.0（Phase 1 二级审批） | 生成日期：2026-06-28

---

## 一、业务流程图

```
员工登录 → [领用申请] → 选择闲置资产 → 填写领用信息 → 提交
                                                          ↓
                                             req_order.status = 0（待部门审批）
                                                          ↓
                           [审批管理] ← 部门管理员通过 → status = 1（待资产管理员审批）
                                                          ↓
                           [审批管理] ← 资产管理员通过 → status = 2（已通过）
                                              ↓               ↓
                                         驳回 → status = 3   资产: 闲置→在用, 绑定使用人
                                                              ↓
                                          [归还登记] ← 使用完毕归还
                                              ↓               ↓
                                         完好 → status = 4    资产: 在用→闲置
                                         损坏 → status = 4    资产: 在用→维修
```

## 二、数据库表结构

### 2.1 req_order（领用申请单）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT UNSIGNED | 主键（自增） |
| apply_no | VARCHAR(30) | 申请编号：`RY-YYYYMMDD-XXXX`（唯一） |
| asset_id | BIGINT UNSIGNED | 领用资产ID → `asset_info.id` |
| applicant_id | BIGINT UNSIGNED | 申请人ID → `sys_user.id` |
| applicant_dept_id | BIGINT UNSIGNED | 申请人部门ID → `sys_dept.id` |
| purpose | VARCHAR(500) | 领用用途（10-500字符） |
| expect_duration | VARCHAR(20) | 预计时长（1/3/6/12个月/自定义） |
| expect_return_date | DATE | 预计归还日期 |
| **status** | TINYINT UNSIGNED | **0-待部门审批, 1-待资产管理员审批, 2-已通过, 3-已驳回, 4-已归还** |
| remark | VARCHAR(200) | 申请人备注 |
| return_date | DATE | 实际归还日期 |
| return_asset_status | TINYINT | 归还完好状态：0-完好, 1-有损坏 |
| return_damage_desc | VARCHAR(500) | 损坏说明 |
| return_remark | VARCHAR(200) | 归还备注 |
| version | INT UNSIGNED | 乐观锁版本号 |

### 2.2 req_approval_log（审批日志）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT UNSIGNED | 主键 |
| requisition_id | BIGINT UNSIGNED | 领用单ID → `req_order.id` |
| approver_id | BIGINT UNSIGNED | 审批人ID → `sys_user.id` |
| approval_level | TINYINT | 审批级别：1-部门管理员, 2-资产管理员 |
| approval_result | TINYINT | 审批结果：0-驳回, 1-通过 |
| reject_reason | VARCHAR(200) | 驳回原因（10-200字符） |

### 2.3 资产状态联动矩阵

| 环节 | req_order.status | asset_info.status | asset_info.user_id |
|------|:---:|:---:|:---:|
| 提交申请 | 0 | 0 闲置 | NULL |
| 部门审批通过 | 1 | 0（不变） | NULL |
| 资产管理员通过 | 2 | **0→1 在用** | **= applicant_id** |
| 任何驳回 | 3 | 0（不变） | NULL |
| 归还-完好 | 4 | **1→0 闲置** | NULL |
| 归还-损坏 | 4 | **1→3 维修** | NULL |

### 接口 1：提交领用申请

```
POST /api/requisition/apply
Content-Type: application/json

Request:
{
  "assetId": 54,              // Long, 必填 *  资产ID
  "purpose": "领用用途说明",    // String, 必填 *  10-500字符
  "expectDuration": "3个月",   // String, 必填 *  1/3/6/12个月 or 自定义
  "expectReturnDate": "2026-12-31", // Date, 必填 *  不早于今天
  "remark": "备注"             // String, 可选  最多200字符
}

Success: 200  "领用申请已提交，请等待审批"
Error: 400 "该资产当前不可领用"
Error: 400 "您已对该资产提交过待审批的领用申请，请勿重复提交"
Error: 400 "归还日期不能早于当前日期"
Error: 400 "操作繁忙，请稍后重试"（分布式锁）
```

### 接口 4：审批通过

```
PUT /api/requisition/approval/pass
```

## 七、Postman 测试用例集

### 7.1 正常领用-审批-归还全流程

```bash
# 前置：登录获取 token
TOKEN=$(curl -s -X POST http://localhost:8080/api/login \
  -H "Content-Type: application/json" \
  -d '{"username":"admin","password":"Eams@123456"}' | grep -o '"token":"[^"]*"' | cut -d'"' -f4)

# 1. 提交领用申请（先用 asset/list 查一个闲置资产ID）
curl -s -X POST http://localhost:8080/api/requisition/apply \
  -H "Authorization: Bearer $TOKEN" -H "Content-Type: application/json" \
  -d '{"assetId":54,"purpose":"Testing requisition system for ten char","expectDuration":"3 months","expectReturnDate":"2026-12-31"}'
# → 200 "领用申请已提交，请等待审批"

# 2. 部门审批通过（status 0→1）
curl -s -X PUT http://localhost:8080/api/requisition/approval/pass \
  -H "Authorization: Bearer $TOKEN" -H "Content-Type: application/json" \
  -d '{"requisitionId":1}'
# → 200 "审批通过"

# 3. 资产管理员终审通过（status 1→2，资产→在用）
curl -s -X PUT http://localhost:8080/api/requisition/approval/pass \
  -H "Authorization: Bearer $TOKEN" -H "Content-Type: application/json" \
  -d '{"requisitionId":1}'
# → 200 "审批通过"

# 4. 归还列表
curl -s http://localhost:8080/api/requisition/return/list?pageNum=1 \
  -H "Authorization: Bearer $TOKEN"

# 5. 确认归还（完好）
curl -s -X PUT http://localhost:8080/api/requisition/return \
  -H "Authorization: Bearer $TOKEN" -H "Content-Type: application/json" \
  -d '{"requisitionId":1,"returnDate":"2026-06-28","returnAssetStatus":0}'
# → 200 "资产已归还"
```

## 九、联调常见坑点

| # | 问题 | 原因 | 处理 |
|---|------|------|------|
| 1 | 审批通过报 "审批结果不能为空" | `@Valid` 在 Controller `setApprovalResult` 前校验 | 已移除 `@NotNull` 注解 |
| 2 | 管理员首次通过 status=0 报 "请等待部门管理员先审批" | 原逻辑 `isAssetAdmin` 不处理 status=0 | 已修复：管理员可处理两级 |
| 3 | StatusTag 显示与 DB 状态不对应 | 旧 REQ_MAP 仅 4 值（0-3），DB 有 5 值（0-4） | 已更新为 5 值模型 |
| 4 | curl 中文乱码导致 JSON 解析失败 | Windows Git Bash 编码问题 | 用 ASCII 字符测试，前端浏览器正常 |
| 5 | 前端 `expectReturnDate` 传空 | 非自定义时长时前端 `computeReturnDate()` 计算 | 后端不依赖前端传值 |
| 6 | 归还后资产状态未更新 | 事务 `@Transactional` 确保原子性 | 联调时验证 asset_info 状态变化 |
